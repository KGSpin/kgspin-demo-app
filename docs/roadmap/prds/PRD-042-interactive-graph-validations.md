# PRD-042: Interactive Graph Validations (HITL Feedback Loop)

**Status:** Draft
**Owner:** VP of Product, CEO
**Date:** 2026-03-03

## High-Level Vision
Currently, the KGenSkills extraction engine uses automated deterministic scripts to generate "Gold" data for bundle tuning. The CEO has proposed moving toward a Human-in-the-Loop (HITL) system where end-users can directly interact with the knowledge graphs built in the demonstrator UI to provide explicit feedback signals. 

This provides a continuous stream of verified False Positive (FP) and False Negative (FN) data directly back into the Tuning Loop, accelerating bundle evolution.

## Proposed User Experience

### 1. Flagging False Positives (The KGS Graph)
When a user is viewing a graph generated by the KGS engine (e.g., in Tab 1 "KGS vs Gemini" or Tab 2 "Ticker Intelligence"):
- **Interaction:** The user clicks on an extracted node or edge that they believe is incorrect (e.g., "worldwide vice president" classified as an ORGANIZATION).
- **Action:** A "Flag Extraction" modal opens.
- **Data Displayed:** The modal displays the full lineage of that extraction (the sentence evidence, the source document, the chunk ID, and the confidence score).
- **User Input:** The user can select a reason for the flag (e.g., "Incorrect Entity Type", "Hallucinated Relationship", "Contextually Wrong").
- **Result:** The system logs this specific extraction as a structurally verified **False Positive** for the specific domain bundle.

### 2. Flagging False Negatives (The LLM Graph)
When a user is viewing a graph generated by the Gemini/Baseline LLM:
- **Interaction:** The user clicks on an edge that the LLM successfully extracted, but which the KGS engine *missed* (revealing a gap in our fingerprint coverage).
- **Action:** A "Save as Gold Data" modal opens.
- **User Input (Hallucination Proofing):** Because LLMs frequently hallucinate "clean" evidence sentences to justify their extractions, the user *cannot* simply click a blind "Confirm" button. They must explicitly validate the extraction:
  1. **Predicate Selection:** The user must select the correct relationship predicate from a dropdown list constrained strictly to the active Bundle's defined schema (e.g., `leads`, `competes_with`). 
  2. **Evidence Citation:** The user must highlight or validate the actual, verbatim evidence sentence from a snippet of the source document displayed in the UI. 
- **Result:** The system logs this specific extraction as a structurally verified **False Negative** (a "Confirmed Missed" gold fact) for the specific domain bundle, anchored to real text rather than LLM hallucinations.

## API Requirements
To support this UX, we need to expose two new endpoints on the FastAPI server:

1.  `POST /api/feedback/false_positive`: Accepts a payload containing the extraction `id`, `evidence_sentence`, `source_document`, `subject`, `predicate`, `object`, and the user's `reason`. Appends this data to an active tuning dataset.
2.  `POST /api/feedback/false_negative`: Accepts a payload containing the `evidence_sentence`, `source_document`, `subject`, `predicate`, and `object`. Appends this data to the active "Gold Data" ledger.

## Architectural Decisions

1.  **Storage Mechanism (SurrealDB Staging):** Feedback signals will *not* be appended directly to the one-off deterministic JSONL files, nor will they be injected live into an actively running tuning loop (which could cause catastrophic disruption). Instead, they will be stored in a dedicated **Staging Table** within SurrealDB (PRD-001). 
    - The `kgen_tune.py` script will be modified to pull an atomic snapshot of this staging table at the *start* of Iteration 1. 
    - No new HITL data will be introduced mid-flight during iterations 2+.
2.  **Lineage-Informed Tuning:** User interactions capture rich metadata from a lineage perspective (e.g., exactly which bundle rule generated the False Positive). The `kgen_tune.py` script will be updated to leverage this structural lineage. By passing this context to the LLM Editor agent, it can make highly informed mutations regarding the validity of a specific rule or suggest new rules to handle edge cases without breaking existing logic.
3.  **Authentication/Abuse:** The demonstrator will remain internal/private for now. Therefore, we do not need to build out RBAC or abuse-prevention mechanisms for these API endpoints in the MVP.
4.  **Entity-First Precedence (Tuning Loop Filter):** To prevent compounding errors and redundant rule mutations, `kgen_tune.py` MUST implement a filter when loading the `hitl_fp` snapshot. If an Edge is flagged as a False Positive, the tuning script must check if either the Subject or Object node is also in the `hitl_fp_entities` set (e.g., due to a `not_a_proper_noun` error). If it is, the Edge-level error must be **suppressed**. The Editor Agent must only receive the Entity-level failure so it can prioritize fixing extraction boundaries before mutating relationship logic.
